# ADR-007 — VaultAPI Domain Model & PolicyTicket v1 (Freeze Point)

**Status:** Accepted · **Task:** P0-09

## Context
ARCHITECTURE.md §3.2 designates `VaultModel` (realized here as the `VaultAPI` package) as the domain schema for persons, sections/fields, history lists, relationships, and provenance, and §3.3 describes the `PolicyTicket` handshake ("a signed, short-lived, operation-scoped token"). ROADMAP.md's M0 freeze-point table lists this package: Track B (AutofillEngine/IngestionPipeline) and Track C (VaultManagerUI/PrivacyDashboard) both build against these types once landed, so — per ADR-006's identical precedent for `PDFEngineAPI` — a further change to what's frozen here is a breaking-contract event requiring a superseding ADR, not a silent edit.

## Decision
Freeze the following as VaultAPI v1 (`Packages/VaultAPI/Sources/VaultAPI/`):

- **Field-path system:** `FieldSection` (the ten PRD FR-2.1 sections + `custom`), `FieldPath` (validated, dot-separated, lowercase; `FieldPath.custom(_:)` is the FR-2.1 "custom fields/sections" mechanism; `FieldPath.isPrefix(of:)` lets a `PolicyTicket` grant a whole section). The concrete catalog of paths lives in `docs/specs/vault-schema.md`, not in this package — `FieldPath` enforces shape, the doc owns the "what fields exist" fact (CLAUDE.md §10's one-owner-per-fact rule).
- **Value types:** `FieldValueKind`/`FieldValue` (string/date/number/enum/list per FR-2.2 — `.enumeration`'s raw value is `"enum"` to keep the wire vocabulary matching the PRD despite the Swift keyword collision), `SensitivityTier` (standard/sensitive, FR-2.5), `Provenance`/`ProvenanceRegion` (manual | document+page+region+confidence).
- **Profiles & relationships:** `PersonID`, `PersonKind`, `Person` (FR-2.3's person/organization profiles), `RelationshipKind`/`RelationshipEdge` (directed, not auto-mirrored).
- **History lists:** `DateRange`, `HistoryCategory` (address/employer/education/travel), `HistoryFieldEntry`, `HistoryEntry` — first-class per ARCHITECTURE.md §8.2, not folded into flat `ProfileField`s, because gap-detection queries need date ranges.
- **Field storage shape:** `ProfileField` (person + path + value + sensitivity + aliases + verifiedAt + provenance; `type` is derived from `FieldValue`'s case rather than stored redundantly, unlike the conceptual DB row in §8.2 — avoids a desync invariant a real store would otherwise have to maintain by hand).
- **`PolicyTicket`:** operation-scoped (`VaultOperation`: read/write/compareRead/cryptoShred — collapsed to one `write` case rather than full CRUD granularity, since PolicyKit mints a ticket per user-visible decision, not per SQL verb), time-boxed (`issuedAt`/`expiresAt`, half-open at expiry), field-path-scoped (`scopedPaths` + `covers(_:)`), with an opaque `signature: Data` — minting and verifying the signature is PolicyKit's job, entirely out of this package's scope.
- **`VaultClient` protocol:** person CRUD, field CRUD, `compareRead` (conflict-detection summary via `FieldSummary`, never the raw value — ARCHITECTURE.md §5.1's "compare-only grant"), history-entry CRUD, relationship CRUD, `cryptoShred` (FR-2.6). Every operation but `lockState()` takes a `PolicyTicket` — there is no protocol-level bypass (CLAUDE.md §3.3).
- **`FakeVaultClient`:** an in-memory `actor` implementing `VaultClient`, enforcing the same structural ticket checks (operation/person/path-scope/expiry) a real implementation must, shipped in the library per CLAUDE.md §5's `Fake*` convention.
- **`VaultConformanceSuite`:** conformance checks shipped in the library (not `Tests/`), so `VaultStore`'s future test target can run the identical suite against the SQLCipher-backed implementation (P1-08's stated acceptance criterion).
- **`SecureBytes`:** see "SecureBytes decision" below.

## SecureBytes decision
Constitution Art. 11 (immutable) and CLAUDE.md §7.3/§5 require decrypted vault values to travel as `SecureBytes`, never bridged to `String` except at the final UI/engine write. The task's Requirements list (as filed) did not name this type, but per root CLAUDE.md's own precedence rule ("nothing in a task file may override a Security or Privacy rule in this document") and §7.9 ("if you find a security defect, fix or file it immediately as Sev-1; do not note it and move on"), shipping `FieldValue.string(String)` on a frozen seam would have been exactly the defect those rules exist to prevent — retrofitting it later means a breaking, superseding ADR for every downstream consumer.

Scope of the fix, deliberately narrow:
- `FieldValue.string` carries `SecureBytes` — the freeform-text vector (names, SSNs, passport/license numbers, addresses) that is the actual "decrypted secret" concern.
- `.number`/`.date`/`.enumeration` stay plain Foundation/`String` types: they're structured or closed-set values, not freeform text, and this codebase has no `SecureNumber`/`SecureDate` precedent to build from. If a future task needs a stronger guarantee there, that's an additive change, not a breaking one.
- `Person.displayName` stays plain `String`. A profile's display name is lower-sensitivity than its field values (comparable to a document title — it's the label the profile picker/UI shows, not itself a passport number or SSN), and every `VaultClient` method that returns a `Person` still requires a `.read`-scoped `PolicyTicket` for that person, so it isn't exposed without any grant at all. Flagging this explicitly for reviewer override: if this reasoning doesn't hold, it's a small, additive follow-up (`Person.displayName: SecureBytes`), not a seam-breaking one, since nothing downstream has been built against it yet.
- `SecureBytes` here is a **wire/DTO-shape**, not a memory-hardening primitive: it forces every hop through the one, greppable `exposeAsPlaintext()` seam and redacts `description`/`debugDescription` (so accidental `print`/interpolation/logging can't leak a value), but it has no `deinit`-driven zeroization or `mlock`-backed storage (Swift value types have no deinit; VaultAPI is Foundation-only per `Scripts/import-allowlist.txt`, so it cannot reach `mlock`/Security-framework primitives without a new dependency, which is its own ADR). The actual hardened master-key handling (`mlock`ed, zeroized on lock) is `VaultStore`/`Platform`'s job per P1-08 and is unaffected by this decision.

## Consequences
- Any change to a protocol signature or a value type listed above is a frozen-seam change: requires a superseding ADR + `[INTEGRATION]`-marked PR with human review (root CLAUDE.md §3.6/§21) — not a normal task diff.
- `VaultClient` requiring a `PolicyTicket` on person-level reads (`person(_:)`, `relationships(for:)`), not just field-level reads, is stricter than the task's Background quote ("read fields {passport.number} for fill-session S") literally shows, but matches CLAUDE.md §3.3's absolute wording ("every privileged vault operation requires a PolicyTicket... there is no bypass path"). A future task that finds this too strict for a specific low-sensitivity read is a scoped, additive follow-up, not a retraction of this ADR.
- `FieldValue`'s `stableFingerprint()` (backing `compareRead`'s `FieldSummary.valueFingerprint`) is a dependency-free FNV-1a hash, not a cryptographic one (CLAUDE.md §17 disallows new dependencies here; a `CryptoKit`-backed alternative would need its own ADR). It is documented as a comparison-only contract, not a security control — a real `VaultStore` implementation is free to use a stronger hash internally as long as the same-value/different-value contract holds, which is what `VaultConformanceSuite.verifyCompareRead` actually tests.
- Vault-locked behavior (`VaultLockState`) has no protocol-level setter — locking is a biometric/password flow the real `Vault.xpc` owns — so `VaultConformanceSuite` cannot exercise it generically; that negative test lives in `FakeVaultClient`'s own behavior tests instead (`FakeVaultClientBehaviorTests.testLockedVaultRejectsEveryOperation`), using the fake-only `setLockState(_:)` seeding method.
