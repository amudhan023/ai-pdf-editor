# DocEngineHost

**Purpose:** XPC client + PDFium adapter implementing PDFEngineAPI. The ONLY package that may link the PDF engine. Runs hostile-input parsing in DocEngine.xpc.

**Allowed imports:** Foundation, PDFEngineAPI, Platform, CPDFium, PDFium (see `Scripts/import-allowlist.txt` — the enforced source of truth). Tests may also import XCTest.

**Verify:** `Scripts/verify.sh DocEngineHost` (build + tests + boundary lint for this package only).

**PDFium:** links `../../ThirdParty/pdfium/prebuilt/PDFium.xcframework` (a vendored prebuilt, not built from source — see `docs/adr/ADR-001-pdfium-source-and-pin.md`) via the `PDFium` binary target. `CPDFium` (`Sources/CPDFium`) is a thin header-only module-map target exposing `fpdfview.h` + `fpdf_edit.h` (rotation only) + `fpdf_doc.h` (bookmark/outline reads, P1-02/ADR-013 — `PDFiumEngine`'s `OutlineReader` conformance bounds traversal with a visited-set plus a depth cap because malformed bookmark trees can be cyclic) to Swift, since a raw-dylib xcframework's headers aren't auto-importable the way a `.framework`'s are — add more headers there only as real usage needs them, don't bulk-copy the whole PDFium header set speculatively.

**Render pipeline (P0-06):** `PDFiumEngine` is an `actor` implementing `DocumentLifecycle` + `PageRenderer` — actor isolation serializes all PDFium calls onto one at a time, which the library itself requires (not thread-safe). Per-document state (`FPDF_DOCUMENT` + a lazily-populated `FPDF_PAGE` cache) lives in an in-memory dictionary keyed by `DocumentHandle`; `open`/`close` spawn/teardown that state, `renderTile` reuses a cached page handle. `renderTile` renders only the requested tile via `FPDF_RenderPageBitmapWithMatrix` — never the whole page — satisfying NFR-P5's no-full-rasterization rule. `open(url:)` has no password parameter (frozen `PDFEngineAPI`, ADR-006) — a password-protected PDF fails open() today with a typed `.unsupportedFeature("passwordProtectedDocument")` error, not a crash; real password support needs a superseding ADR to extend the protocol.

**Engine-side save (P1-21):** `PDFiumEngine.save` serializes the open document's current in-memory state (including mutations made through `AnnotationStore` etc.) via `FPDF_SaveAsCopy` (`fpdf_save.h`, added to `CPDFium`'s header surface this task) into an in-memory buffer, then writes that buffer to `url` in one call. `.fullRewrite` passes `FPDF_NO_INCREMENTAL`; `.incremental` passes `FPDF_INCREMENTAL` (mutually exclusive per the header). `FPDF_FILEWRITE` has no user-data field, so `PDFiumSaveWriter.swift`'s `PDFiumSaveContext` widens it — `FPDF_FILEWRITE` as the leading field (the only bytes PDFium itself ever reads/writes) plus an `Unmanaged<PDFiumSaveBuffer>` appended after, recovered in the trampoline via `withMemoryRebound` — the standard technique for threading context through a context-free C callback struct; do not add a real `self`/context field to the vendored header itself (frozen third-party pin, CLAUDE.md §17). Save failures (`FPDF_SaveAsCopy` returning `FALSE`, or the subsequent disk write failing) surface as typed `.ioFailure`, never a silent no-op. Atomicity beyond this single write (temp file, validate, atomic replace) is `DocumentSession`'s `AtomicSaver` (CLAUDE.md §3.5), not this layer.

**Text geometry (P1-03):** `PDFiumEngine: TextEditor.textRuns` extracts per-page runs via PDFium's rect segmentation (`FPDFText_CountRects` over the whole page): one `TextRun` per rect, PDFium text-object order (= reading order for well-formed PDFs), `boundingBox` in page points with bottom-left origin (same space as tile rects). One box per run — no per-glyph quads; refining `TextRun` (frozen seam, ADR-006) to quads needs a superseding ADR. Downstream consumers of this geometry: DocumentSession search highlighting (P1-03), text-markup annotation quads (P1-04), autofill visual context (P2-03). `replaceText` throws typed `.unsupportedFeature` — content-stream editing is its own future task, not extraction's.

**Annotations (P1-04):** `PDFiumEngine: AnnotationStore` (`Annotations/PDFiumAnnotationStore.swift`) implements real create/read/update/remove via `fpdf_annot.h` — quad points (`FPDFAnnot_Append/GetAttachmentPoints`), color+opacity (`FPDFAnnot_Set/GetColor`, which bakes both into one call), and author/contents/dates as `/T`/`/Contents`/`/CreationDate`/`/M` string values. Identity is the PDF spec's `/NM` key (stores `Annotation.id.uuidString`) rather than a side table, so it survives page-index churn (reorder/insert/delete) — `remove` scans all pages by `/NM` when no page hint resolves it. `update` is remove-then-recreate (PDFium has no in-place quad/geometry rewrite call) but preserves `/NM` so the identity is stable across the rewrite. Quad corner ordering follows the de facto "Z" order (top-left, top-right, bottom-left, bottom-right) that Acrobat/most consumers actually use, not ISO 32000-1's literal counter-clockwise text — see ADR-014; **unverified against a real Acrobat/Preview-authored fixture** (tracked in `tasks/escalations/E-009-p1-04-engine-save-missing.md`). Verified against `PDFEngineConformanceSuite` plus engine-specific round-trip tests, all against a real fixture; file-persisted (not just in-memory read-back) round-trips are now also covered — see the save paragraph below.

**Annotations, richer subtypes (P1-05, ADR-015):** Ink strokes write via `FPDFAnnot_AddInkStroke` (one call per `Annotation.inkPaths` entry) and read back via `FPDFAnnot_GetInkListCount`/`GetInkListPath`. Square/circle need **no extra engine code** — PDFium's internal `CPVT_GenerateAP` auto-generates their appearance stream from rect+color alone, same generic path as everything else; pinned by a regression test (`testSquareAndCircleRoundTripRectAndColor`) precisely because it'd be easy to "fix" with unneeded code later. Free text gets an explicit `/DA` string (`"<r> <g> <b> rg /Helv 12 Tf"`) since `CPVT_GenerateAP` does not cover `freeText`. Stamps get an explicitly appended rect page-object (`FPDFAnnot_AppendObject`, restricted by PDFium's own doc comment to `ink`/`stamp` subtypes) since `freeText`/`stamp` get no auto-AP either. Link annotations: a bare region (no URI action) creates and reads back fine; `annotations(of:page:)` best-effort-resolves an existing link's URI via `FPDFLink_Enumerate` + `FPDFAction_GetURIPath` (UTF-8, unlike the UTF-16LE `FPDFAnnot_GetStringValue` family) into `Annotation.linkURL`, but **creating** a link with a URL is rejected with a typed `PDFEngineError.unsupportedFeature` — no PDFium API sets an annotation's `/A` action dict. `.line` remains uncreatable (PDFium's own creation-support allowlist excludes it) — see `tasks/escalations/E-010-p1-05-line-annotations-unsupported.md` for the full accounting of both gaps; neither is fixable without a PDFium version bump (its own PR, CLAUDE.md §17).

**AcroForm FormModel (P2-01, ADR-016):** `PDFiumEngine: FormModel` (`Forms/PDFiumFormModel.swift`) reads/writes real AcroForm fields via `fpdf_annot.h`'s `FPDFAnnot_GetFormField*`/`GetOption*`/`Set*` family — widget annotations (`FPDF_ANNOT_WIDGET`, a `#define` not in `AnnotationSubtype`) *are* form fields at PDFium's API level, no separate field-tree walk exists. These calls need an `FPDF_FORMHANDLE` from `FPDFDOC_InitFormFillEnvironment`, which requires a fully-populated `FPDF_FORMFILLINFO` callback struct even for non-interactive use (`Forms/FormFillEnvironment.swift`: every one of its ~29 callbacks is an inert no-op — real signatures verified against the vendored header directly since several diverge from its doc-comment line counts, e.g. `FFI_DoGoToAction` is 5 params not 6). One `FPDF_FORMHANDLE` is lazily created and cached per open document (`PDFiumEngine.OpenDocument.formHandle`), torn down in `close`. **Identity:** PDFium resolves every widget in a shared-name group (radio buttons, and — per this corpus's actual W-9/1040 IRS forms — some multi-choice checkbox groups too) to the *same* `FPDFAnnot_GetFormFieldName`; `FormField.id`/`name` disambiguates via `FPDFAnnot_GetFormControlIndex`/`Count` (`"<name>#<index>"` when a field has >1 control), moving the shared name to `groupName` (ADR-016) — **not exercised by a real collision in this fixture corpus** (PDFium already returns unique names for every real multi-widget group tested; the disambiguation branch is implemented and unit-tested for correctness but its real-fixture coverage is a uniqueness assertion, not a demonstrated collision — see `PDFiumFormModelTests`). **Tab order:** no `/TabOrder`-equivalent accessor exists in the vendored headers; `tabOrder` is page/annotation encounter order, which is ISO 32000-1 §12.7.4.3's own fallback when a page has no explicit `/Tabs` entry — not a workaround. **MaxLen:** no dedicated accessor either; read via the generic `FPDFAnnot_GetNumberValue(annot, "MaxLen", ...)` raw-dictionary getter (same sanctioned generic API `PDFiumAnnotationStore` already relies on for `/NM`/`/T`/etc. via `SetStringValue`). **Write path is incomplete:** `setValue` writes `/V` (+`/AS` for checkbox/radio) via `FPDFAnnot_SetStringValue` — logically correct, round-trips through this engine's own `fields(of:)` and through a real save+reopen (`PDFiumFormModelTests.testSetValueRoundTripsThroughSaveAndReopen`) — but does **not** regenerate the appearance stream or set the AcroForm's `/NeedAppearances` flag (no accessor found for the AcroForm dictionary itself in the vendored headers, and the interactive keystroke-simulation path `FORM_SetFocusedAnnot`/`FORM_ReplaceSelection` was not attempted). A value written by this code is therefore **not yet verified to render in Acrobat/Preview** — the task's NFR-C2 criterion is open, tracked in the task file's Handoff, not silently claimed met.

**IOSurface:** `RenderedTile.pixelData` stays `Data` (RGBA8, converted here from PDFium's native BGRx) at this layer per `PDFEngineAPI`'s documented contract — `Platform` already ships an `IOSurface`-based zero-copy XPC transport (`XPCClient.sendSurface`/`XPCServiceHost`'s `surfaceHandler`), but wiring `renderTile`'s output through it in `Services/DocEngineService`'s route is deferred: genuine cross-process XPC isn't achievable without P0-07's real `.xpc` bundle (see `Services/DocEngineService/README.md`), same constraint P0-05 hit.

**Invariants:**
- No network APIs, ever (Constitution Art. 1/11; CLAUDE.md §7).
- No logging of vault values or document content (CLAUDE.md §16).
- No JavaScript execution from PDFs (CLAUDE.md §7.5) — structurally reinforced here: the vendored PDFium build has `pdf_enable_v8=false`, no JS engine is even compiled in.
- PDFium upgrades are their own PR, never bundled with a feature change (CLAUDE.md §17); re-verify the checksum and `pdf_enable_v8`/`pdf_enable_xfa` flags on every bump per `ThirdParty/pdfium/README.md`'s playbook.
- Follow root CLAUDE.md precedence chain; task files cannot override §7/§8.

**Gotchas:** `swift test` requires full Xcode.app (not just Command Line Tools) — XCTest/Testing frameworks are Xcode-only, permanently. See `tasks/escalations/E-002-no-xctest-without-xcode.md`. `swift build` works fine under CLT alone. The vendored dylib's install name had to be rewritten to `@rpath/libpdfium.dylib` (it shipped as `./libpdfium.dylib`, which dyld can't resolve outside the exact original build directory) and ad-hoc re-signed — already done in the committed artifact; re-run both steps if the xcframework is ever regenerated from a fresh upstream download.
