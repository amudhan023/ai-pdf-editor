# DocumentSession

**Purpose:** Document lifecycle: open/edit/atomic save/backups, undo stack, viewer + annotation + form-fill UI. Application layer.

**Allowed imports:** Foundation, PDFEngineAPI, Platform (see `Scripts/import-allowlist.txt` — the enforced source of truth). Tests may also import XCTest.

**Verify:** `Scripts/verify.sh DocumentSession` (build + tests + boundary lint for this package only).

**Invariants:**
- No network APIs, ever (Constitution Art. 1/11; CLAUDE.md §7).
- No logging of vault values or document content (CLAUDE.md §16).
- Follow root CLAUDE.md precedence chain; task files cannot override §7/§8.

**Markup annotations (`Sources/DocumentSession/Annotate/`, P1-04):** `AnnotationUndoStack` is a pure value type (no engine access) — `record(_:)` takes a `Change` (`.added`/`.removed`/`.updated(before:after:)`), `undo()`/`redo()` return the *action to perform* (not just "what happened"), so `DocumentSession` stays the only thing that talks to the engine. `DocumentSession.annotations/addAnnotation/updateAnnotation/removeAnnotation/undoAnnotation/redoAnnotation` wrap an optional `annotationStore: AnnotationStore?` — unwired degrades to `[]` reads and a typed `.engine(.unsupportedFeature)` throw on writes, same optional-capability contract as `outlineReader`/`textEditor`. `MarkupToolbarViewModel` (`@MainActor` view model, mirrors `SearchViewModel`'s pattern) and `MarkupToolbarView` (mirrors `SearchBarView`) drive the toolbar. **Documented scope cut:** creation targets one caller-supplied `TextRun`, not a multi-run drag selection — there's no drag-to-select gesture in the viewer (P1-03 built search highlighting over existing runs, not interactive text selection); the wired affordance is "Mark Selection" on the current search hit. Existing markup is click-to-select via `PageTileView`'s annotation overlay (`onSelectAnnotation`), enabling delete/undo/redo — no resize/move/reshape editor yet. **Not met by this task, tracked via `tasks/escalations/E-009-p1-04-engine-save-missing.md`:** file-persisted round-trip and Acrobat/Preview interop-fixture corpus, both blocked on `DocEngineHost`'s `save` being unimplemented (P1-21).

**Richer annotations (P1-05, ADR-015/E-010):** `MarkupToolbarViewModel.pickerSubtypes` grew from 4 text-markup subtypes to 9 (adds note/`.text`, free text, square, circle, stamp) plus a published `selectedOpacity`, all reusing the same `TextRun`-rect creation flow — only `quadSubtypes` (highlight/underline/strikeOut/squiggly/link) attach `/QuadPoints`; everything else uses `boundingBox` alone. `.ink` and `.link` are deliberately **not** in the picker: ink needs a real freehand-stroke capture gesture (none exists in the viewer) and a rect-only ink annotation would be an invisible empty `/InkList`; a toolbar-placed link has no meaningful URI to attach and the engine can't set one anyway. Both remain fully engine/session-tested via `DocEngineHost`, just not toolbar-reachable this pass. **Comment sidebar (`CommentSidebarViewModel`/`CommentSidebarView`):** a third `DocumentSidebarView` pane listing every `.text`-subtype annotation document-wide (author/contents/created date), select navigates via the same `DocumentViewModel.navigate(to:)` path as thumbnails/search, delete removes through the session. Reply-free v1 per the task's own wording — no threaded-reply model exists in `PDFEngineAPI` to build on. **Not met by this task:** move/resize/reshape UI for any annotation subtype (still no editor, same gap as P1-04), and the `<8ms` ink-drawing latency acceptance criterion is unverified — no freehand drawing gesture exists yet to measure, and this environment can't run a live trackpad-input bench.

**Save wiring (P1-22):** `DocumentSession.save(mode:)` tracks the URL it was opened with, calls `DocumentLifecycle.save(_:mode:to:)` to a same-directory sibling temp file (same volume, so `AtomicSaver`'s replace is a true atomic rename — never system `/tmp`, and the file only exists for the duration of the call), then commits through the existing `AtomicSaver` (P1-16) write-temp -> validate -> atomic-replace path. `atomicSaver` is an optional init param (same degradation contract as `outlineReader`/`textEditor`/`annotationStore`) — unwired throws `.engine(.unsupportedFeature)` rather than silently no-opping. Engine failures surface as `.engine(...)`; atomic-replace failures as the distinct `.saveFailed(AtomicSaveError)` case — neither is swallowed or generalized. Tested via `MockPersistingEngine` (test-local, genuinely round-trips annotation bytes through disk), not real `PDFiumEngine`: this package's import allowlist structurally forbids depending on `DocEngineHost`'s concrete engine (composition-root-only, see `App/CLAUDE.md`) — the equivalent real-engine round trip already lives in `DocEngineHost`'s own P1-21 tests.

**Gotchas:** `swift test` requires full Xcode.app (not just Command Line Tools) — XCTest/Testing frameworks are Xcode-only, permanently. See `tasks/escalations/E-002-no-xctest-without-xcode.md`. `swift build` works fine under CLT alone.

**Tiling architecture (`Sources/DocumentSession/Viewer/`, P1-01):** `TileGrid` is pure geometry (page points -> grid-aligned `PDFRect`s intersecting a viewport + prefetch margin); `TileCache` is an actor, LRU-evicted by a total-byte budget (not entry count — payload size varies a lot with scale) with a `respondToMemoryPressure` call-in point the app wires to `DispatchSource.makeMemoryPressureSource` (the source's handler fires off-actor, so the actor can't own it directly). `DocumentViewModel.tile(page:tileRect:scale:)` is cache-first: a miss renders through `DocumentSession` and populates the cache. `ZoomMath` is pure scale/anchor arithmetic backing `ZoomMode` (`.fitPage`/`.fitWidth`/`.custom`); `ScrollPosition`/`ScrollPositionStoring` persist page-granularity (not sub-page fraction) reopen position. Known scope cut: within-page visible-rect culling isn't wired into the SwiftUI view (`PageTileView` renders a page's full tile grid once any part of the page is on-screen) — only page-level virtualization via `LazyVStack`; true sub-page culling would need an AppKit `NSScrollView` bridge for continuous scroll-offset access, out of this pass's scope.

**Sidebar (`Sources/DocumentSession/Sidebar/`, P1-02):** `ThumbnailSelectionModel` is a pure value type with macOS list semantics (click/⌘-toggle/⇧-range-from-anchor) — selection is keyed by *positional* `PageIndex`, so P1-06's drag-reorder must remap or `clear()` it after any page mutation (see the type's doc comment). `DocumentViewModel` owns the shared sidebar/viewer state: `outline` (loaded once per open through `DocumentSession.outline()`; a failed read degrades to empty, never fails the open), `currentPage` (viewer reports via `pageDidBecomeVisible`), and `navigationTarget` (UUID-wrapped so repeat clicks on the same page still fire SwiftUI `onChange`). Thumbnails render one full-page tile at scale 0.15 — deliberately the same scale as `PageTileView`'s placeholder, so both share one `TileCache` entry per page; virtualization is `LazyVStack` row laziness plus the cache's byte budget.

**Search (`Sources/DocumentSession/Search/`, P1-03):** `DocumentTextSearcher` streams hits page-by-page (`AsyncThrowingStream`; first results land before later pages are scanned — that's the <300ms-first-result bar, asserted structurally in tests) and cancels cooperatively at page boundaries. Matching is `SearchTextNormalizer.fold`: NFKC (ligatures ﬁ→fi) then case/diacritic/width folding — script-neutral, RTL included. Hit granularity is the whole `TextRun` (frozen-seam single box): `snippet` = run text, highlight = run box, drawn by `PageTileView`'s overlay with the same Y-flip transform as tiles. `SearchViewModel` owns incremental query state (each keystroke cancels + restarts), wrap-around next/previous, and navigates through the same `DocumentViewModel.navigate(to:)` path as the sidebar. `DocumentSession.textRuns(page:)` returns `[]` when no `textEditor` is wired (same degradation contract as `outline()`).
